# Kubernetes

Короткий довідник основних елементів архітектури Kubernetes.

## Control Plane

Керує кластером, зберігає його стан і вирішує, де запускати навантаження.

- **API Server** - відкриває Kubernetes API і приймає запити від користувачів, утиліт та компонентів кластера.
- **Controller Manager** - запускає контролери, які порівнюють бажаний стан із реальним і виправляють різницю.
- **Scheduler** - вибирає Node для нового Pod з урахуванням ресурсів, обмежень, affinity і taints.
- **Cloud Controller Manager** - підключає Kubernetes до хмарних API для балансерів, дисків, маршрутів і вузлів.
- **etcd** - узгоджена key-value база, яка зберігає стан кластера, об'єкти, конфігурацію і метадані.

## Worker Node

Запускає Pod-и застосунків і дає локальний runtime, мережу та агент вузла.

- **Node** - робоча машина, фізична або віртуальна, на якій Kubernetes запускає Pod-и.
- **Kubelet** - агент вузла, який створює Pod-и, стежить за їхнім станом і звітує в API Server.
- **Kube Proxy** - налаштовує мережу вузла, щоб трафік Service доходив до правильних Pod-ів.
- **Container Runtime** - програма, яка завантажує образи і запускає контейнери, наприклад containerd або CRI-O.
- **Pod** - найменша одиниця запуску в Kubernetes, зазвичай з одним app container і helper containers.
  - **Container** - ізольований процес у Pod, створений з образу і підключений до мережі Pod.
  - **Probes** - health checks, за якими Kubernetes бачить, чи контейнер живий, готовий або ще стартує.
    - **Liveness Probe** - перевіряє, чи контейнер живий; при провалі Kubernetes перезапускає його.
    - **Readiness Probe** - перевіряє, чи контейнер готовий до трафіку; при провалі прибирає його з Service.
    - **Startup Probe** - дає повільним контейнерам більше часу перед liveness і readiness перевірками.
  - **Resources** - налаштування CPU і пам'яті, які допомагають планувати та обмежувати контейнери.
    - **Requests** - мінімальні CPU або пам'ять, зарезервовані для контейнера і враховані Scheduler.
    - **Limits** - максимальні CPU або пам'ять, які контейнер може використати.

## Services

Дають стабільну мережу для Pod-ів, IP-адреси яких можуть змінюватися.

- **Service** - стабільна віртуальна точка, яка надсилає трафік до відповідних Pod-ів або заданих endpoint.
- **ClusterIP** - стандартний внутрішній тип Service, доступний тільки всередині кластера.
  - **Headless Service** - Service без віртуального IP, який повертає DNS записи окремих Pod-ів.
- **NodePort** - відкриває Service на однаковому порту кожної Node для зовнішнього доступу.
- **ExternalName** - прив'язує внутрішнє ім'я Service до зовнішнього DNS імені.
- **LoadBalancer** - просить хмарного провайдера створити зовнішній балансер і вести трафік до Service.

## Controllers

Тримають workloads близько до бажаного стану: створюють, замінюють, оновлюють або видаляють Pod-и.

- **ReplicaSet** - підтримує фіксовану кількість однакових Pod-ів.
- **Deployment** - керує ReplicaSet і дає rollout, rollback, scaling та оновлення stateless застосунків.
- **StatefulSet** - запускає stateful Pod-и зі стабільними іменами, порядком старту і постійним сховищем.
- **DaemonSet** - запускає копію Pod на кожній вибраній Node, часто для логів, метрик або мережі.
- **Job** - виконує одноразове завдання, доки потрібні Pod-и не завершаться успішно.
- **CronJob** - створює Job за розкладом, подібно до Linux cron.
- **Horizontal Pod Autoscaler** - автоматично змінює кількість Pod-ів за метриками, як CPU або memory.
- **Ingress Controller** - читає Ingress об'єкти і маршрутизує зовнішній HTTP або HTTPS трафік до Services.

## Configuration

Відділяє налаштування застосунку і чутливі значення від container images.

- **ConfigMap** - зберігає несекретні налаштування як key-value пари, файли або змінні середовища.
- **Secret** - зберігає чутливі дані для Pod-ів; base64 саме по собі є кодуванням, не шифруванням.

## Storage Objects

Підключають Pod-и до тимчасового або постійного сховища.

- **Volume** - сховище, змонтоване в контейнери Pod; часто живе стільки ж, скільки Pod.
- **PersistentVolume** - ресурс сховища в кластері, створений вручну або динамічно через StorageClass.
- **PersistentVolumeClaim** - запит на розмір і режим доступу сховища, який прив'язується до PersistentVolume.
- **StorageClass** - описує тип сховища і те, як Kubernetes має динамічно створювати PersistentVolume.

## App Types

Пояснюють, чи залежить застосунок від стабільної ідентичності та постійних даних.

- **Stateless** - застосунок без локального стану; Pod-и можна вільно замінювати або масштабувати.
- **Stateful** - застосунок, якому потрібні стабільна ідентичність, порядок роботи і постійні дані.

## Ingress

Керує зовнішнім HTTP і HTTPS доступом до Services.

- **Ingress Controller** - Сам ресурс кубернетес який виконує маршрутизацію пакетів в самому кластері.
- **Ingress Resource** - правила маршрутизації, які зіставляють hosts і paths із Services у кластері.

## Scheduler Rules

Керують тим, які Node підходять для запуску Pod-ів.

- **Taint** - мітка на Node, яка відштовхує Pod-и без відповідної toleration.
- **Toleration** - дозвіл на Pod запускатися на Node з відповідним taint.
- **Affinity** - правило, яке притягує Pod до певних Node або ближче до певних Pod-ів.
- **Anti-Affinity** - правило, яке тримає Pod далі від певних Node або далі від певних Pod-ів.

## RBAC

Role-Based Access Control

- Subjects
  - **User** - особа, аутентифікація якої відбувається поза K8s.
  - **Group** - Об'єднання користувачів у групи.
  - **Service Account** - ідентифікатор для програм (подів), що працюють всередині кластера.
- RBAC Components
  - **Role** - набір дозволів на дії з ресурсами, обмежений одним *Namespace*.
  - **Cluster Role** - набір дозволів, що діє на рівні всього кластера (або для не-іменних ресурсів, як-от Node).
  - **RoleBinding** - прив'язка *Role* до суб'єкта у межах конкретного *Namespace*.
  - **ClusterRoleBinding** - прив'язка Cluster Role до суб'єкта, що надає права доступу до ресурсів у всьому кластері.

## CRD

- **CRD (Custom Resource Definition)** - схема, яка описує ваш власний тип об'єкта, розширюючи стандартний API Kubernetes.
- **CR (Custom Resource)** - конкретний екземпляр об'єкта, створений на основі ```CRD```.

## HELM

- **Charts** - пакунок з усіма ресурсами (маніфестами), необхідними для запуску програми в K8s.
- **Releases** - конкретний екземпляр чарту, розгорнутий у кластері.
- **Values** - Значення які підставляються у темплейт, вони знаходяться в Values.yaml

## API Gateway

Аналогічно до Ingress слугує як єдина точка входу в кластер, відповідає за термінацію TLS, балансування навантаження. На відміну від Ingress має більший набір функціоналу і працює переважно на рівні аплікації (L7)

Щоб створити API Gateway потрібно створити наступні ресурси:

- **Gateway Class** - (Хто) Показує який саме контроллер буде обслуговувати API Gateway.
- **Gateway** - (вхід, порти, TLS) Ентріпоінт в кластер, описує які порти буде слухати Gateway. На цьому етапі термінується TLS. Щоб його зазначти потрібно чітко вказати який StorageClass буде обслуговуват цей Gateway.
- **HTTP Route** - (правила, фільтри, бекенди) Відповідає за марштутизацію трафіку за чітко визначеними правилами в цьому маніфесті. Один Gateway може мати декілька HTTP Route якщо їх всіх зазначити до цього Gateway.
