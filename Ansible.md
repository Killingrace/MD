# Ansible

Ansible — це безкоштовний інструмент для автоматизації IT-процесів, керування конфігураціями та розгортання софту (Infrastructure as Code / Configuration as Code).

## Ansible Work Principle

Ansible працює за принципом push-моделі: ви запускаєте інструмент на своєму комп'ютері або керуючому сервері (`Control Node`), він зчитує написаний вами сценарій (`Playbook`) та список цільових серверів (`Inventory`), а потім підключається до цих серверів напряму через стандартний протокол SSH або WinRM. Після підключення Ansible копіює на віддалені машини невеликі програми-модулі, виконує їх там для налаштування системи або встановлення софту і після завершення роботи автоматично видаляє ці тимчасові файли, не залишаючи в системі зайвого сміття чи фонових процесів. Програми конфігурації залишаються на `Control Node`, оскільки вони є частиною Ansible.

Основні елементи в Ansible

- **Playbook** - це головний файл конфігурації (сценарій), в якому описано що і на яких саме серверах потрібно зробити

- **Roles** - це незалежний, модульний набір завдань, змінних, шаблонів та файлів, який структурований в окрему папку і присвячений одній конкретній темі.

- **Tasks** -  це найменша, атомарна одиниця дії в Ansible. Вона викликає один конкретний інструмент (модуль) для виконання однієї конкретної операції на сервері.

- **Inventory** - Текстовий файл (зазвичай у форматі INI або YAML), де прописані IP-адреси або доменні імена всіх керованих серверів, розбитих за групами (наприклад, [webservers], [dbservers]).

- **Control Node** - Комп'ютер або сервер, на якому встановлено сам Ansible і з якого запускаються всі команди та сценарії.

- **Managed Nodes** - Цільові сервери, хости або мережеві пристрої, якими керує Ansible (на них самих Ansible не встановлюється).

- **Modules** - Готові блоки коду (плагіни), які виконують конкретні завдання на серверах: встановлюють пакети (apt, yum), копіюють файли (copy), керують службами (systemd).

- **Handlers** - Спеціальні завдання, які спрацьовують лише тоді, коли інше завдання внесло якісь зміни (найчастіше використовуються для перезапуску сервісів після зміни їхніх конфігураційних файлів).

Структура:

```sh
Playbook -> Role -> Task
```

```sh
my_ansible_project/
├── ansible.cfg            # Глобальні налаштування Ansible для цього проекту
├── inventory/             # Папка з файлами хостів (серверів)
│   ├── production.ini     # Сервери для продакшену
│   └── staging.ini        # Сервери для тестування
├── group_vars/            # Змінні для груп хостів
│   ├── all.yml            # Змінні, доступні для всіх серверів
│   ├── webservers.yml     # Змінні лише для групи webservers
│   └── dbservers.yml      # Змінні лише для групи dbservers
├── host_vars/             # Змінні для конкретних серверів (за їх іменами/IP)
│   ├── prod-web-01.yml
│   └── stage-db-01.yml
├── roles/                 # Папка з перевикористовуваними ролями
│   ├── webserver/         # Роль для налаштування веб-сервера (напр. Nginx)
│   │   ├── tasks/
│   │   │   └── main.yml   # Головні завдання ролі
│   │   ├── handlers/
│   │   │   └── main.yml   # Трігери (наприклад, перезапуск сервісу після змін)
│   │   ├── templates/
│   │   │   └── nginx.conf.j2 # Шаблони конфігів (Jinja2)
│   │   ├── files/
│   │   │   └── index.html # Статичні файли для копіювання
│   │   ├── vars/
│   │   │   └── main.yml   # Внутрішні змінні ролі (високий пріоритет)
│   │   └── defaults/
│   │       └── main.yml   # Дефолтні змінні ролі (низький пріоритет)
│   └── dbserver/          # Інша роль (наприклад, для PostgreSQL)
│       └── tasks/
│           └── main.yml
├── site.yml               # Головний плейбук, який запускає все
├── webservers.yml         # Плейбук суто для веб-серверів
└── dbservers.yml          # Плейбук суто для баз даних
```

## Configs

 Ansible використовує Jinja2 для створення конфігураційних шаблонів. Файли конфігурації мають розширення `.j2` і значення із плейбуку підславляють у `{{ var }}` структуру.

Використовуються темплейти у модулі `ansible.builtin.template:`

Приклад коду:

```yaml
ansible.builtin.template:
    owner: root
    group: root
    src: index.j2
    dest: /var/www/html/index.html
    mode: 777
    vars:
        greeter: Hello
```

## Error Handling

 В Ansible обробка помилок (Error Handling) — це суперважлива тема. За замовчуванням Ansible працює жорстко: якщо якась таска на сервері падає з помилкою, він повністю зупиняє виконання плейбука для цього сервера і переходить до наступного (якщо він є).

Але в реальному житті нам часто треба ігнорувати дрібні помилки, перевіряти специфічний текст у логах або виконувати "відкат" (rollback).

- **block, rescue, always** - try, except, finally
- **failed when** - додатковий рейз для фейлу плейбуку
- **ignore errors** - ігнорування помилок під час виконання таски.
- **any_errors_fatal** - зупинка всього плейбуку якщо хоч один сервер зупиниться із помилкою.
- **force_handlers** - виклик хендлерів навіть попри помиклу у виконанні таски.

## Vaults

Ansible Vault — це вбудований інструмент, який дозволяє шифрувати чутливі дані (паролі, токени, SSH-ключі, сертифікати) прямо в твоїх файлах конфігурації, інвентарях або змінних. Зашифровані файли можна безпечно зберігати в Git.

Для шифрування використовується стійкий алгоритм AES-256.

- **ansible-vault create \<filename>** - створити Vault.
- **ansible-vault encrypt \<filenane>** - зашифрувати вже готовий файл із даними.
- **ansible-vault decrypt \<filenane>** - розшифрувати готовий зашифрований файл.
- **ansible-vault view \<filenane>** - переглянути секрети із хранилища
- **ansible-vault edit \<filenane>** - змінити дані.
- **ansible-vault rekey \<filenane>** - змінити ключ доступу.
