# Terraform

Terraform - це`IaC` інструмент для керування інфраструктурою, надає можливість описувати декларативно або імперативно яку інфраструктуру ми хочемо отримати. Terraform використовує мову `HCL` (Hashicorp Configuration Language).

## IaC

**IaC** (Infrastructure as Code) - практика, за якою вся проектова інфраструктура описується у вигляді коду. Це може бути як локальне середовище так і великі дата центри. Такий підхід допомагає легко реплікувати та версіонувати інфраструктуру. Також надає можливість виправляти відхилення у середовищих (`Envinromental Drift`), завдяки чому інфраструктура не буде відхилятись від зазначеного курсу.

**Environmental Drift** - Розходження між реальним станом інфраструктури та станом, який описаний в Terraform та зафіксований у файлі конфігурації.

## Source Conrtol For Infrastructure

За допомогою Terraform ми отримуємо систему контролю версій, оскільки інфраструктура представляє собою код.\
Можливості:

- **Change History** - надає можливість перегляду історію всіх архітектурних змін та просто ідентифікувати які зміни були здійснені та коли

- **Consistency and Stability** - надає консистентність конфігурації всіх середовищ, від розробки до проду. Зменшує ризик людського фактору та маленьких змін, які зможуть зламати те чи інше середовище

- **Team Collaboration** - коли приймється практика IaC, то інфраструктура слідкується системою контролю версій, тож кожен член команди може працювати над змінами незалежно, забезпечуючи ефективну роботу та співпрацю.

- **Automation and Recovery** - легка інтеграція із CI/CD пайплайнами. Можна керувати інфраструктурою в пайплайнах.

- **Managing State** - Terraform допомагає управляти станом інфраструктури, зберігаючи станові файли, які трекають кожен ресурс змінений або створений Terraform. З цими файлами Terraform може робити роллбеки до минулих станів системи.

## Terraform Objects

- **Terraform File** - файл із конфігураціями Terraform, має закінчення **.tf**.
- **Terraform Project** - Директорія або папка, що об'єднує один або декілька файлів Terraform.
- **Resources** - Основні елементи які будуть створені чи керовані Terraform.
- **Providers** - Плагіни, які дозволяють Terraform взаємодіяти із ресурсами різних хмарних провайдерів (Azure)
- **Input Variables** - Змінні, що передаються до конфігурації під час розгортання інфраструктури. Можуть мати локальне значення, або приймати параметки із cli або envvars.
- **Locals** - локальні, значення, які не можна змінити ззовні. Використовуються всередині модуля, щоб уникнути дублювання коду.
- **Module** - це контейнер для коду, яки дозволяє зручно код упаковувати та використовувати. Модулі можуть перевикористовуватись. Також модулям можна централізовано змінювати конфігурацію із головного файлу, змінюючи **vars**.
Модулі можна писати власноруч, а можна завантажувати із Terraform Module Registry.

- **Privisioners** - це інструменти, які дозволяють виконувати додаткові команди або дії після створення ресурсу. Існує декілька видів

  - `local-exec` - Запуск коду на локальній машині.

  - `remote-exec` - Запуск команди на ресурсі. Для використання потрібно вказати додаткові дані для підклюення до ресурсу, наприклад по **SSH**

  - `file` - Передача файлу на ресурс. Для використання потрібно вказати додаткові дані для підклюення до ресурсу, наприклад по **SSH**

- **Data Sources** - дозволяють отримувати інформацію про наявні ресурси з провайдера. Це можуть бути ресурси, створені іншими користувачами, або ресурси, створені поза Terraform.

## Files

- **Specific:**
  - **.terraform.lock.hcl** - файл, головна ціль якого забезпечити використання того самого провайдера на всіх машиних, які запускатимуть Terraform код
  - **.terraform** - директорія, в якій Terraform зберігає завантажені бінарники модулів та провайдерів.
  - ***.tfstate** - файл у якому Terraform зберіє поточний стан інфраструктури.
  - **terraform.tfvars** - файл конкретних значень для variables.
- **Project Structure:**
  - main.tf
  - providers.tf
  - variables.tf
  - outputs.tf
  - locals.tf

## Terrafirn Ecosystem

- **tfsec** - Статичний аналізатор коду, спирається на Best Practices безпеки від провайдерів. Створює список вразливостей, які можна відразу виправити.

- **trivy** - Допомагає знайти вразливості та неправильні конфігурації у коді, контейнерних образах, файлових системах та Git репозиторіях.

- **Terraset** - Призначений для автоматизації тестування інфраструктури та дозволяє перевірити розгорнуті конфігурації, забезпечуючи їхню правильність.

## Language

- **count** - Cтворення nнної кількості ресурсів за `count` індексом.

- **foreach** - Cтворення nнної кількості ресурсів вказуючи потрібні параметри.

- **Lifecycle** - Cпеціальний блок конфігурації, який дозволяє вам змінювати стандартну поведінку Terraform під час створення, модифікації або видалення конкретного ресурсу., можуть підвищини відмовостійкість інфраструктури під час оновлень, наприклад `create_before_destroy`, `prevent_destroy`, `ignore_changes`.

- **dynamic** - автоматичне створення повторюваних частин коду. Сам код вказується в блоці `content`.

- **functions** - вбудовані функції Terraform, які спрощують роботу із кодом. наприклад `lower`, `upper`, `split`, `distinct`

## State

Terraform зберігає дані стану архітектури у файлі .tfstate. він може зберігатись як локально так і віддалено.

Бекенд вказується в `terraform/backend`. Види бекенду:

- **Local** - зберігається локально у директорії проекту або за вкзаним шляхом.
- **Remote** - зберігається у віддаленому сховищі. Щоб вказати бекенд, наприклад в Azure Storage Account потрібно вказати: storage_account_name, container_name, key(filename), resource_group_name.

Для ініціалізації бекенду потрібно скористатись з `terraform init`, а щоб мігрувати бекенд потрібно вказати `terraform init -migrate-state`

Щоб перевірити список ресурсів потрібно написати `terraform state list` а щоб дізнатись, що терраформ знає про окремий ресурс потрібно написати `terraform state show <azuremrm_resource_group.example>`.

>Коли починається розгортання інфраструктури, Terraform накладає Lease (блокування) на .tfstate файл у віддаленому бекенді. Таким чином не можна паралельно двічі запустиити `terraform apply`.

## Workspaces

Можливість Terraform використовувати один і той самий код для декількох середовищ (Dev/Staging/Prod).

Workspace створюється командою `terraform workspace new <WorkspaceName>`.
Перемикання відбувається через `terraform workspace іудусе <WorkspaceName>`

Після перемикання між Workspace, Terraform використовує той самий код, але створює окремий, ізольований **.tfstate** файл.
