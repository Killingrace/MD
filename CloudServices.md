# Cloud Computing Basics

Хмарні технології — це концепція використання великої кількості обчислювальних ресурсів, які знаходяться на віддалених серверах та доступні через інтернет. Приклади: `AWS`, `Azure`, `GCP`

- **Server** - це окремий комп’ютер, який використовується для запуску застосунку, в якого багато користувачів.
- **Data Center** — окрема будівля, яка спеціально обладнана для того, щоб забезпечувати безперебійну роботу і підключення до інтернету великої кількості серверів
- **Cloud Provider** - компанії, які спеціалізуються на побудові та утриманні дата-центрів.

## Clouds

Хмари

- **Public Cloud** - Можливість використовувати інфраструктуру і сервіси, що належать хмарному провайдеру, доступ до яких здійснюється через інтернет. Цьому варіанті у нас немає потреби утримувати власний датацентр.
- **Private Cloud** - Коли компанія утримує власний датацентр, але при цьому датацентр надає послуги внутрішнім користувачам компанії, точно так само як ці сервіси надаються в публічній хмарі. В таких випадках використовують `DCOS`. Використоувється лише компаніями яким необхідна велика кількість серверів, або коли компанії зобов'язані законом тримати дані в певній локації. Складність в масштабовуванні
  - **DCOS** - Комплексний програмний продукт який встановлюється на всі сервери дата центру та розподіляє навантаження між ними, наприклад вирішує на якому саме запустити віртуальні машину
- **Hybrid Cloud** - Поєднання приватної та публічної хмари, що дозволяє організаціям оптимізувати свої ресурси та підвищити гнучкість. Можемо використовувати публічну хмару для додатків які не вимагають високого рівня безпеки та приватності, а приватну хмару для критичних застосунків. Дозволяє за потреби просто масштабувати ресурси у публічній хмарі. Забезпечує безпеку та контроль над критично важливими даниим в приватній хмарі. Мережеве підключення між хмарами може відбуватись через `VPN` або через виділені лінії з'єдання, такі як `Azure Express Route`, забезпечуючи безпечний мережевий тунель для даних.

### Cloud Benefist

Переваги хмари

- Зменшення витрат для малих та середніх фірм
- просте масштабування за потребою
- Простий доступ до нового обладання та нових сервісів
- Уже влашновані інструменти моніторингу та захисту
- Сприяє швидкості розробки та впровадження нових продуктів

## Azure Architecture

Архітектура Azure

### Physical Infrastructure

Дата центри, розподілені на регіони, які в свою чергу розподілені на зони покриття. Існує багато різних регіонів які достосовуються під вимоги та закони даної території, на якій знаходиться дата центр. Регіони з'єднані швидким підключенням, що надає можливість додаткам між дата центрами в межах одного регіону комунікувати майже без затримок.

### Management Infrastructure

Будівельні блоки із яких будується власна інфраструктура, тобто середовище роботи своїх застосунків.

- **ARM API (Azure Resource Manager)** - Azure сервіс який керує створенням, оновленням, конфігурацією та виділенням ресурсів.
- **Azure Account** - опубліковий запис в Azure.
  - **Entra ID** - хмарна служба керування доступами (SaaS), яка забезпечує автентифікацію користувачів, захист облікових даних та централізований контроль доступу до хмарних ресурсів.
  - **Tenant** - Особистий Ізольований екземпляр хмари в Azure.
- **Management Group** - дозволяють групувати підписки в ієрархічній структурі.
- **Azure Subscription** - підписка, яка працює як єдиний центр управління, де можна керувати доступом до сервісів та відслідковувати витрати. Підписка відкриває доступ до продуктів на послуг Azure. Один акаунт Azure може мати декілька підписок.
  - **Azure Policy** - Правила, із якими будуть створені власні ресурси, за допомогою `Azure Policy`. Політики допомагають запроваджувати стандарти побудови інфраструктури та моніторитиїх дотримання.
- **Resource Group** - один або декілька ресурсів, поєднана належністю до однієї групи ресурсів. При видаленні Групи видаляються всі залежні ресурси. Кожна ресурсна група обов'язково має бути створена в межах однієї підписки `Azure Subscription`.
- **Resource** - Все що створюється або розгортається в Azure вважається ресурсом. Основний будівельний блок. При створенні має бути зазначений до однієї конкретної `Resource Group`.

## Services

### IaaS - Infrastructure as a Service

Модель, в якій хмарний провайдер відповідає лише за фізичні сервери та датацентри, а ти за роботу OS, потрібних тобі програм та даних називається `IaaS`(Infrastructure as a Service). Загалом до IaaS відносять всі сервіси які імітують роботу фізичного обладнання.

Приклади `IaaS`

- Virtual Machines
- Virtual Network
- Storage Services

### PaaS - Platform as a Service

`Pass` (Platform as a Service) - це модель сервісу яка надає тобі вже готову платформу для встановлення твоїх програм. PaaS бере на себе відповідальність за керування OS та середовищем на якому запускається програма. Тобі залишається лише передати хмарному провайдера код застосунку.

- Azure Web Apps
- Azure Functions (Serverless Computing, paid by number of running)
- DB Services (MSSQL, MySQL, PostgreSQL, CosmosDB)
- Azure Blob Storage (low cost storage service, paid by storage volume)
- Azure Kubernetes Service (Control Plane Managed by Azure)

### SaaS - Software as a Service

`SaaS`(Software as a Service) - модель, в якій ти просто купуєш у хмарного провайдера готовий сервіс. Хмарний оператор бере на себе відповідальність за OS, платформу, застосунок. Єдина твоя відповідальніть це твої дані.

- Azure App insights (Monitoring Service)
- Microsoft Entra ID (User Account Management)
- Azure DevOps
- GitHub

### SLA

SLA (Service Level Agreement) - контракт, який указує рівень послуги, що надається сервісним провайдером. Кожен ресурс в Azure має SLA.

В контексті VM вказуються в SLA

- **Unavailable** статус в контексті VM. VM вважається недоступною тоді, коли вона не може з'єднатись із іншими сервісами або неможливо приєднатись до цієї VM.
- **Max available minutes** - час, протягом якого VM увімкнена користувачем.
- **Downtime** - Час, протягом якого віртуальна машина була недоступна в момент планованої активності.
- **Uptime** - Відсоток часу роботи віртуальної машини протягом якого VM працювала та користувач оплатив її роботу

В більшості сервісів Azure гарантує Uptime 99.9% - 99.99%, якщо умова не дотримується, користувач отримує `Service Credit` (компенсацію) 10% - 100% від загальної вартості послуги.

### Disks

Disk types services

- **Unmanaged disk**
- **Managed disk**

Storage types

- **Premium** (SSD)
- **Standard** (HDD)

Replication types. Види реплікації даних. Реплікація відбувається асинхронно із мінімальною затримкою, де один накопичувач (БД) записує а дві інші дублюють, а читання відбувається одночасно із трьох накопичувачів.

- **LRS** (Locally-Redundant Storage) - Реплікація тричі в межах одного датацентру Azure на різних серверах
- **ZRS** (Zone-Redundant Storage) - Реплікація тричі в межах різних датацентрів Azure на різних серверах в зазначених зонах. Збільшує вартість диску
- **GRS** (Geo-Redurant Storage) - Реплікація тричі в межах трьох регіонів. Найдорожча опція.

Disk Roles

- OS Disk - Завантаження OS
- Data Disk - Збереження даних окремо від OS
- VM Temporary Disk - Збереження даних поки VM існує. Не відображається як окремий диск.

### Azure Monitor

Ресурс, головна ціль якого моніторити задані ресурси шляхом збору відповідних метрик, які цілком охарактеризовіють стан моніторованого ресурсу на момент зробу метрик.

- **Metrics**
  - **Built-in** - (Host-Level Metrics) Базові метрики, які віддає моніторований ресурс. (CPU Usage, I/O, Network)
  - **Custom** - (OS-Level) Кастомні метрики, створені користувачем. Або задаються вручну або створюються за допомогою `Azure Resource Agent`. Платні. (Memory Usage, Disk Usage, etc)
- **Metrics storage** - База даних для збереження метрик зібраних із моніторованого ресурсу. Автоматично створюється із Azure Monitor. Дані дійсні 93 дні.
- **Azure Resource** - Моніторований ресурс, що віддає метрики до Azure Monitor. За замовчуванням має набір вбудованих метрик.
- **Metrics Explorer** - Інструмент для аналізу та візуалізації метрик в Azure. Доступний як окремий сервіс.
- **Log Analytics** - Інструмент для збору і аналізу логів. Платний.
- **Alerts** - Алерти, можуть створюватись як для метрик так і для логів. Платний.

### Service Backup

Сервіс який автоматично робить бекапи для потрібного ресурсу. Всі бекапи зберігає в `Backup Storage`.

- **Backup Store** - Хранилище для збереження бекапів
- **Backup Policy** - Налаштування яке містить тип ресурсу, розклад створення, та `Retention Policy`
- **Retention Policy** - Інформація про кількість резервних копій та час зберігання.
- **Backup Job** - процес створення бекапу для кожного ресурсу.
  - **Crash Consistent** - Збереження поточного ресурсу без урахування цілісності даних. Не впливає на продуктивність системи.
  - **File Consistent** - Бекап із урахуванням цілісності файлів. Потребує фонову програму, що замородує OS перед створенням бекапу.

## Identity and Access Management (IAM)

Керування ідентифікаторами та доступами в Azure здійснюється за допомогою **Microsoft Entra ID** (раніше відомого як Azure Active Directory або Azure AD). Це хмарна служба керування ідентифікаторами та доступом.

### Identities

Ідентифікатори — це об'єкти, які можуть бути автентифіковані в системі та яким можна надати доступ до ресурсів.

- **Users (Користувачі)** - Облікові записи для реальних людей. Можуть бути внутрішніми (створеними безпосередньо в Entra ID) або гостьовими (B2B, запрошеними з інших організацій або поштових сервісів).
- **Services** - Managed Identity, Service Principal
- **Groups (Групи)** - Логічне об'єднання користувачів, сервісних суб'єктів або інших груп. Використовуються для спрощення адміністрування: замість видачі прав кожному користувачу окремо, права видаються групі.
  - **Security Groups** - Використовуються виключно для керування доступом до ресурсів.
  - **Microsoft 365 Groups** - Включають доступ до ресурсів, а також спільну поштову скриньку, календар та SharePoint.

### Service Accounts & Application Identities

Ідентифікатори, призначені для програмного коду, скриптів та автоматизації, щоб вони могли безпечно взаємодіяти з ресурсами Azure без використання особистих даних реальних людей.

- **App Object** - Інстанс об'єкту, який описує додаток, автоматично створється в тенанті після виконання `App Registration`.

- **Service Principal** - Локальне представлення (екземпляр) глобального додатка у твоєму Entra ID. Працює як "сервісний акаунт" (технічний користувач) для програм або CI/CD конвеєрів (наприклад, Terraform або GitHub Actions). Щоб отримати Service Principal потрібно спочатку зареєструвати додаток (App Registration), описавши додаток в Azure Entra ID.
  - Потребує ручного створення ключів (Client Secret) або сертифікатів.
  - Секрети мають термін дії, їх потрібно безпечно зберігати (наприклад, в Azure Key Vault) та періодично оновлювати (ротувати).

- **Managed Identity** - Керований ідентифікатор, яким повністю опікується сам Azure. Дозволяє ресурсам Azure (наприклад, VM або Azure Functions) безпечно автентифікуватися в інших сервісах без зберігання паролів чи секретів у коді.
  - **System-Assigned** - Створюється для конкретного ресурсу Azure (відношення 1:1). Його життєвий цикл жорстко прив'язаний до цього ресурсу: якщо видалити VM, її System-Assigned ідентифікатор також буде знищено.
  - **User-Assigned** - Створюється як окремий незалежний ресурс в Azure. Може бути призначений багатьом різним ресурсам (відношення 1:багато). Існує незалежно від того, чи існують ресурси, до яких він прикріплений.

### RBAC (Role-Based Access Control)

Система керування доступом на основі ролей. Надає точний контроль над тим, хто має доступ до ресурсів Azure і що саме вони можуть з ними робити. Базується на принципі найменших привілеїв (Principle of Least Privilege).

Процес надання доступу (Role Assignment) складається з трьох обов'язкових компонентів:

- **Security Principal (Суб'єкт безпеки)** - *ХТО* отримує доступ. Це може бути User, Group, Service Principal або Managed Identity.
- **Role Definition (Визначення ролі)** - *ЩО* можна робити. Це колекція дозволів (Actions) та заборон (NotActions). Бувають Built-in (вбудовані, створені Microsoft) та Custom (кастомні, створені користувачем).
  - **Owner** - Повний доступ до всіх ресурсів, включно з правом роздавати доступ іншим.
  - **Contributor** - Дозволяє створювати та керувати всіма типами ресурсів, але забороняє керувати доступами (не може призначати ролі іншим).
  - **Reader** - Дозволяє лише переглядати існуючі ресурси (Read-only).
- **Scope (Область дії)** - *ДЕ* застосовується цей доступ. Ієрархія Azure дозволяє призначати ролі на чотирьох рівнях (роль, призначена на вищому рівні, автоматично успадковується на нижчих):
  - **Management Group** (найвищий рівень)
  - **Subscription** (Підписка)
  - **Resource Group** (Група ресурсів)
  - **Resource** (Конкретний ресурс, наприклад, окрема база даних або VM)
